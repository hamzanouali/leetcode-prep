Reverse K-Group
Common Mistakes -- LC #25
Mistake #1: Reversing without counting first
The Wrong Approach
Start reversing immediately, then try to "undo" if the group has fewer than k nodes. This leads to corrupted pointers and infinite loops.
What should have tipped you off
"If the number of nodes is not a multiple of k then left-out nodes should remain as-is." This means you MUST know the group size before touching anything.
The Fix
Count first, reverse second. Walk k nodes ahead to verify the group is complete. Only then reverse. If fewer than k remain, stop.
Mistake #2: Losing the connection between groups
The Wrong Approach
Reverse each group correctly but forget to connect the tail of the previous group to the new head of the current group. Result: the list breaks into disconnected segments.
What should have tipped you off
After reversing [1,2,3] to [3,2,1], node 1 is now the TAIL. It needs to point to the head of the next reversed group -- but that head doesn't exist yet.
The Fix
Track a groupPrev pointer. After reversing a group, groupPrev.next = new head and update groupPrev to the old head (now the tail). A dummy head serves as the initial groupPrev.
Mistake #3: Not using a dummy head
The Wrong Approach
Try to handle the first group as a special case. This leads to duplicated logic and bugs when k=1 or when the list length equals k exactly.
What should have tipped you off
The very first group's reversal changes the head of the entire list. Any time the head can change, you need a dummy.
The Fix
Create dummy.next = head. Now every group (including the first) has a "previous tail" to connect to. Return dummy.next at the end.
The Mental Model

Think of it as a 3-step loop: 1) Count k nodes ahead (bail if short). 2) Reverse k nodes with prev/curr/next. 3) Wire: groupPrev.next = new head, update groupPrev = old head. Repeat until you can't count k.
