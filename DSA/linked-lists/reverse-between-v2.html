🎯 The Real-Life Analogy
Imagine a deck of cards in a row on a table: A B C D E.
You want to reverse cards at positions 2-4 (B C D).

The trick: put your finger on B. Now repeatedly take the card right after your finger and place it at the front of the group:

Start: A [B C D] E
Move C to front: A [C B D] E
Move D to front: A [D C B] E

Your finger (B) never moved. It just slid to the end naturally as cards were inserted before it.

beforeLeft (bL) = the gatekeeper. It always points to the current front of the reversed section. Every new node gets inserted right after it.

curr = the original first node of the section. It never moves in memory — but it sinks to the back as nodes are placed ahead of it. At the end, it's the last node of the reversed section.
🧠 The 3 Lines — What & Why
Line 1: curr.next = nodeToMove.next
What: Removes nodeToMove from its current position by skipping over it.
Why: You're about to relocate this node. If you don't detach it first, you'd create a cycle — nodeToMove would be in two places at once.
Bonus: This also connects curr to whatever comes after nodeToMove, keeping the rest of the list intact.
Line 2: nodeToMove.next = beforeLeft.next
What: Points nodeToMove to whatever is currently at the front of the reversed section.
Why: This is how nodeToMove "joins" the reversed section. It gets placed before everything already reversed.
Key detail: beforeLeft.next is the current front — NOT curr. On the first iteration beforeLeft.next happens to be curr, but on later iterations it's a previously moved node.
Line 3: beforeLeft.next = nodeToMove
What: Makes nodeToMove the new front of the reversed section.
Why: beforeLeft is the gatekeeper — the entry point to the reversed zone. Updating it completes the insertion.
Why last? If you did this before Line 2, you'd lose the reference to the old front. The order matters!
⚠️ Why The Order Matters
Each line depends on values that the next line is about to overwrite:

Line 1 reads curr.next (= nodeToMove) before anything changes
Line 2 reads beforeLeft.next (= old front) before Line 3 overwrites it
Line 3 overwrites beforeLeft.next — safe now, Line 2 already used it

Swap Line 2 and Line 3? nodeToMove.next would point to itself (infinite loop!).
Swap Line 1 and Line 2? nodeToMove.next gets overwritten before you detach — you lose the tail of the list.
❓ Questions You Probably Have
Q: Why doesn't curr move forward each iteration?
Because curr isn't walking through the list — it's an anchor. The node right after curr keeps changing (that's nodeToMove), so curr naturally "sees" a new node each time without moving itself. Think of it as standing still while people in line jump over you.
Q: How does this avoid the "reconnection" step that other solutions need?
Other approaches reverse the sub-list and then stitch it back. This approach never breaks the connection between Zone A and Zone B, or between Zone B and Zone C. Every iteration, the list is valid from start to end. beforeLeft always points into the reversed section, and curr always points to what comes after it. The zones are always connected — they just grow and shrink.
Q: Why (right - left) iterations, not (right - left + 1)?
You have (right - left + 1) nodes in the section, but the first one (curr) is already in place — it just needs to end up last. You only need to move the other (right - left) nodes in front of it. Each iteration moves exactly one node.
Q: What if left = 1? There's no "beforeLeft" node!
That's exactly why we use a dummy node. It becomes beforeLeft when left=1. Without it, we'd need special-case code for when the reversal starts at the head.
🔒 The Invariant (After Every Iteration)
After each loop iteration, these are always true:

1. beforeLeft.next → points to the front of the reversed portion so far
2. curr → still the same node (node 2), now at the tail of the reversed portion
3. curr.next → points to the next node that will be moved (or to Zone C when done)
4. The full list is always connected and valid — no dangling pointers at any point
Check this invariant after each frame in v3 — you'll see it holds every time.
💻 The Code (annotated)
function reverseBetween(head, left, right) { const dummy = { next: head }; // fake node before head let beforeLeft = dummy; // anchor #1 // Navigate: stop right before position 'left' for (let i = 0; i < left - 1; i++) { beforeLeft = beforeLeft.next; } let curr = beforeLeft.next; // anchor #2 (never moves!) // Each iteration: grab node after curr, // detach it, insert it at the front for (let i = 0; i < right - left; i++) { const nodeToMove = curr.next; // who's being relocated curr.next = nodeToMove.next; // detach (skip over) nodeToMove.next = beforeLeft.next; // point to old front beforeLeft.next = nodeToMove; // become new front } return dummy.next; }
Time: O(n)
Space: O(1)
💡 Summary
Two anchors (beforeLeft + curr) that never move.
Three lines (detach, point to front, become front) that repeat.
One invariant — the list is always valid after every iteration.
